# Flag `is_demo` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Marcar las compras y usuarios creados en modo demo con un flag `is_demo` para poder borrarlos luego sin tocar los datos reales.

**Architecture:** `provisionGuestPurchase` gana un tercer parámetro opcional `opts:{isDemo?}`; cuando es demo, incluye `is_demo:true` en el upsert de `course_purchases` y pasa `data:{is_demo:true}` al invitar el usuario. La rama real omite la clave, así que no depende de la migración. Migración + SQL de limpieza como archivos.

**Tech Stack:** Supabase (Postgres), TypeScript, Vitest.

## Global Constraints

- La clave `is_demo` se incluye en el payload del upsert **solo** cuando `opts?.isDemo === true`. Sin opts → la clave NO aparece (default `false` de la columna) → el webhook real no depende de la migración.
- No cambiar el comportamiento de la rama real (webhook guest y logueada).
- `inviteUserByEmail(email, { redirectTo, data })` — `data` va a `user_metadata`.
- Firma nueva EXACTA: `provisionGuestPurchase(session, admin, opts?: { isDemo?: boolean })`.
- Tests unit en `__tests__/`. Antes de cada commit: `npx tsc --noEmit` (0 err) y `npx eslint <archivos>` (0 err); suite completa verde antes del commit final de cada tarea.

Bloque de invitación (paso 2) — construir las opciones y pasar `data` solo en demo:
```ts
  if (!userId) {
    const redirectTo = `${process.env.NEXT_PUBLIC_BASE_URL ?? ''}/auth/callback?next=/reset-password`;
    const inviteOptions: { redirectTo: string; data?: Record<string, unknown> } = { redirectTo };
    if (opts.isDemo) inviteOptions.data = { is_demo: true };
    const { data: invited, error: inviteError } = await admin.auth.admin.inviteUserByEmail(email, inviteOptions);
    if (invited?.user?.id) {
      userId = invited.user.id;
    } else {
      const { data: reFetched } = await admin
        .from('profiles')
        .select('id')
        .eq('email', email)
        .maybeSingle();
      userId = reFetched?.id;
      if (!userId) {
        return { ok: false, reason: `invite-failed:${inviteError?.message ?? 'unknown'}` };
      }
    }
  }
```

Bloque del upsert (paso 3) — incluir `is_demo` solo en demo:

    user_id: userId,

  if (opts.isDemo) purchase.is_demo = true;

- [ ] **Step 5: Crear el SQL de limpieza**

-- Limpieza de datos de prueba creados en modo demo. REVISA antes de ejecutar.
-- Borra las compras demo y los usuarios creados en demo (cascada a profiles y
-- a las compras restantes de esos usuarios).
